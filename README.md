# Understanding Events and Alerts

Security monitoring begins with events generated by systems, applications, and network devices.

Examples include:

* User logins
* Process executions
* File downloads
* Privilege changes
* Network connections

These events are logged and forwarded to security platforms such as SIEM, EDR, or SOAR solutions.

Detection rules analyse incoming data and generate alerts when suspicious activity is identified.

Instead of manually reviewing millions of logs, analysts can focus on a manageable number of alerts requiring investigation.

---

# Alert Lifecycle

The alert lifecycle follows a structured process:

1. Event Occurs
2. Event Logged
3. Log Collected
4. Detection Rule Triggered
5. Alert Created
6. Analyst Investigation
7. Classification
8. Escalation or Closure

This process ensures suspicious activity is reviewed consistently and efficiently.

# Alert Properties

A SOC analyst must understand the information contained within every alert.

| Property     | Purpose                           |
| ------------ | --------------------------------- |
| Alert Time   | Time the alert was generated      |
| Alert Name   | Summary of detected activity      |
| Severity     | Business impact and urgency       |
| Status       | Investigation progress            |
| Verdict      | Analyst classification            |
| Assignee     | Responsible analyst               |
| Description  | Detection explanation             |
| Alert Fields | Technical evidence and indicators |

### Example Alert Types

* Unusual Login Location
* Email Marked as Phishing
* Windows RDP Brute Force
* Potential Data Exfiltration
* Suspicious PowerShell Activity

# Alert Severity Levels

Security alerts are commonly prioritised using severity ratings.

### Low

Informational activity with limited impact.

### Medium

Suspicious activity requiring investigation.

### High

Likely malicious activity with significant risk.

### Critical

Urgent threat requiring immediate attention.

---

# Alert Status Workflow

### New

Awaiting analyst review.

### In Progress

Currently under investigation.

### Closed

Investigation completed.

### Escalated

Transferred to a higher-tier analyst.

# Alert Classification

An important responsibility of a SOC Level 1 Analyst is determining whether an alert represents a genuine threat.

## True Positive

Confirmed malicious activity.

Examples:

* Malware execution
* Credential compromise
* Unauthorized access

## False Positive

Legitimate activity incorrectly flagged by a detection rule.

Examples:

* Approved administrative actions
* Normal business processes

---

# Alert Prioritisation Strategy

When multiple alerts exist simultaneously, analysts should prioritise based on:

## 1. Ownership

Investigate only alerts that are unassigned and not already being handled.

## 2. Severity

Process alerts in the following order:

1. Critical
2. High
3. Medium
4. Low

## 3. Time

Investigate older alerts before newer alerts to minimise attacker dwell time.

# SOC Level 1 Investigation Workflow

## Step 1 – Take Ownership

* Assign alert to yourself
* Change status to In Progress

## Step 2 – Review Alert Details

Analyse:

* Alert description
* Detection logic
* Severity
* Indicators of compromise

## Step 3 – Investigate Context

Determine:

* Affected user
* Hostname
* Device
* Network source
* Related activity

## Step 4 – Validate Suspicion

Review surrounding logs and supporting evidence.

Questions to ask:

* Is this expected behaviour?
* Does it match known attack techniques?
* Are there additional suspicious events?

## Step 5 – Classify

Determine whether the alert is:

* True Positive
* False Positive

## Step 6 – Document Findings

Record:

* Investigation steps
* Evidence reviewed
* Final reasoning
* Recommended actions

## Step 7 – Close or Escalate

Escalate confirmed threats to SOC Level 2 analysts or close benign activity.
